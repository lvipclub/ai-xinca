What are the common BACnet object types for HVAC control?

Controls#BACnet#objects#ASHRAE-135

BACnet Object Types for HVAC Control Systems

BACnet (ANSI/ASHRAE Standard 135) models every physical and logical element of a building automation system as an object with standardised properties, making it fundamentally different from register-based protocols like Modbus. Understanding the object model is essential for specifying interoperable BAS architectures.

### Core Input/Output Objects

The six fundamental BACnet object types map directly to physical I/O points. **Analog Input (AI)** objects represent sensor readings — temperature, pressure, humidity, flow, and CO₂ concentration — and include properties for engineering units, resolution, and COV (Change of Value) increment for subscription-based data delivery. **Analog Output (AO)** objects command actuators and analogue devices (0–10V or 4–20mA), with a Priority_Array that allows multiple control applications to write to the same output at different priority levels. **Analog Value (AV)** objects store setpoints, calculated values, and configuration parameters that have no direct physical I/O binding.

The binary equivalents — **Binary Input (BI)** for contact closures (fan status, filter differential switch, freeze-stat trip), **Binary Output (BO)** for start/stop commands and two-position actuators, and **Binary Value (BV)** for software enables and mode selections — follow the same priority array and status flag conventions.

### Higher-Level Objects for System Operation

Beyond the basic I/O, BACnet defines objects that handle scheduling, trending, and alarming — functions that Modbus and other simple protocols cannot natively support. The **Schedule object** provides weekly schedules with exception-day overrides for holidays and special events, enabling the BAS to manage occupancy transitions without custom code. Each Schedule has a default weekly table and can reference calendar objects for recurring exceptions.

The **Trend Log object** automates historical data collection, buffering samples in the controller's memory and transferring them to archive servers. Configuration includes sample interval, data type (polled or COV-triggered), and buffer size — a typical VAV controller might trend zone temperature, airflow, damper position, and heating valve position at 15-minute intervals with a 48-hour local buffer.

The **Notification Class object** manages alarm routing — defining which devices receive alarms, whether acknowledgements are required, and escalation rules for unacknowledged alarms. A single Notification Class can serve multiple Alarm objects, standardising the site's alarm management strategy.

### Multi-State and Command Objects

For discrete operating modes, **Multi-state Input (MSI)** and **Multi-state Output (MSO)** objects handle enumerated states (Occupied/Standby/Unoccupied, Off/Slow/Medium/Fast). **Command objects** sequence multi-step operations like morning warmup — opening the heating valve, verifying flow, and enabling the fan in a defined order with configurable wait and failure conditions.

🔑 Key Takeaways

  • ✓BACnet models every device as standardised objects with mandatory properties, unlike Modbus which only provides register read/write.
  • ✓The Priority_Array on AO and BO objects enables multiple control applications to write to the same output at different priority levels (1=highest, 16=lowest).
  • ✓Schedule, Trend Log, and Notification Class objects provide native scheduling, data logging, and alarm management without custom code.
  • ✓COV (Change of Value) subscription on AI objects reduces network traffic by only transmitting data when values change beyond a configured increment.
  • ✓A well-designed BACnet object layout with consistent naming conventions dramatically simplifies commissioning and ongoing system maintenance.
